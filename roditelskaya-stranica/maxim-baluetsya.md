**Описание работы метода**

1. Запрос формируется исходя из роли пользователя

   1. employeeId

      1. Для линейного сотрудника берется из токена

      2. Нач офиса и термен могут выбирать сотрудников из списка

   2. officeId

      1. Линейный сотрудник и термен могут выбирать офис на фронте

      2. Для начальника офиса берется из его токена

   3. divisionId

      1. Только начальник и термен могут выбрать отображение определенных направлений

   4. regionId

      1. Берется из токена термена, этот параметр доступен только ему

2. Фронт отправляет запрос get <http://leet.best.atbplugin.tech/schedule/schedule> c access токеном в header

3. Бек проверяет подлинность токена

   1. Если токен неверен - отправляет 1010 ошибку

4. Бек проверяет соответствие введенных параметров роли пользователя

   1. Если какая-то роль использует параметр, который для нее заблокирован - отправляет 1011 ошибку

5. Бек проверяет корректность введенных данных в параметрах

   1. Если запрос некорректен - отправляет соответствующую ошибку

6. Бек получает необходимые данные из бд

7. Бек фильтрует данные на основе параметров

8. Бек отправляет отфильтрованные данные фронту

9. Фронт отрисовывает данные

**Описание работы метода**

1. Фронт отправляет запрос get `/schedule/attendance` с параметрами:

   -  `shiftIds` -- строка с ID смен через запятую

   -  access токен в header

   -  *Когда используется:* при открытии графика офиса на день или графика сотрудника на неделю.

2. Бек проверяет подлинность токена через Auth Service

   -  Токен отсутствует, недействителен или истек -> ошибка 1010

   -  Токен действителен -> извлечь `role` и `employeeId`

      -  `role` -- роль пользователя

      -  `employeeId` -- ID сотрудника (для линейных сотрудников)

      -  `officeId` -- ID офиса (для руководителей офиса)

      -  `regionId` -- ID региона (для территориальных менеджеров)

3. Бек проверяет валидность параметров

   -  ID параметры должны быть целыми ≥ 0 -> ошибки 1006, 1007, 1016, 1017

   -  Если передан `shiftIds`, все значения должны быть числами -> ошибка 1024

4. В зависимости от роли пользователя применяются разные фильтры:

   -  Линейный сотрудник (`ROLE_EMPLOYEE`):

      -  Оставляются только смены, где `employeeId` совпадает с ID сотрудника из токена

      -  Если ни одной смены не осталось -> ошибка 403

   -  Руководитель офиса (`ROLE_OFFICE_MANAGER`):

      -  Оставляются только смены, где `officeId` совпадает с офисом менеджера

      -  Если ни одной смены не осталось -> ошибка 403

   -  Территориальный менеджер (`ROLE_TERRITORY_MANAGER`):

      -  Получение списка офисов региона через Office Service

      -  Оставляются только смены, где `officeId` входит в этот список

      -  Если ни одной смены не осталось -> ошибка 403

5. Бек формирует запрос к БД

   -  Парсит `shiftIds` в массив чисел

   -  Выполняет `SELECT` из таблицы `attendance`

   -  Фильтрация:

      -  По `shift_id`

      -  ИЛИ по связанным сменам, отфильтрованным по параметрам

   -  Учитывает только активные смены (`state = 'scheduled'`)

6. Бек формирует ответ

   -  Создает объект `attendances`

   -  Ключ -- `shiftId`, значение -- объект посещаемости

   -  Каждый объект содержит: `id`, `shiftId`, `status`, `actualStart`, `actualEnd`, `absenceReason`

   -  Возвращает **200 OK**

7. Фронт получает данные

   -  Если запрос был для модального окна -- заполняет поля формы

   -  Если запрос для обновления графика -- обновляет статусы смен

8. Фронт рассчитывает дополнительные статусы смены:

   -  Если есть `factStartAt` и `factStartAt` > `startAt` -> статус "опоздание"

   -  Если есть `factEndAt` и `factEndAt` \< `endAt` -> статус "ранний уход"

   -  Если есть `factStartAt` и `factEndAt` и (factEndAt - factStartAt) > (endAt - startAt) -> статус "переработка"

   -  Если есть и опоздание, и ранний уход -> статус "опоздание + ранний уход"

9. Обновляет иконки и отображение

10. Фронт отображает дополнительные статусы и всплывающую подсказку с фактическим временем

11. Фронт отображает график пользователю

**Описание работы метода**

1. Действия пользователя

   -  Начальник офиса (или система) выбирает несколько дней в календаре

   -  Для каждого выбранного дня выбирает свой шаблон из списка

   -  Нажимает кнопку "Применить шаблоны"

2. Валидация на фронте

   -  Фронт проверяет, что выбран офис (officeId есть)

   -  Фронт проверяет, что массив items не пустой

   -  Для каждого элемента проверяет наличие date и templateId

   -  Фронт проверяет уникальность дат в запросе

   -  Фронт формирует тело запроса

3. Отправка запроса

   -  Фронт отправляет post /schedule/apply-templates с:

      -  JWT access токеном в заголовке Authorization

      -  JSON-объектом с officeId и массивом items

4. Проверка авторизации

   -  Бек проверяет подлинность токена через Auth Service:

      -  Токен недействителен -> 1010 Не авторизован

5. Проверка прав доступа

   -  Бек проверяет права доступа:

      -  Только роль ROLE_OFFICE_MANAGER может применять шаблоны

      -  Иначе -> 1011 Доступ запрещен

6. Контекстная проверка

   -  Бек проверяет контекстную безопасность:

      -  officeId из запроса должен совпадать с officeId менеджера -> иначе 1011

7. Валидация тела запроса

   -  Бек валидирует body запроса:

      -  officeId обязателен, целое число ≥ 1 -> иначе 1006

      -  items обязателен, не пустой массив -> иначе 1050

      -  Для каждого item:

         -  date в формате YYYY-MM-DD -> иначе 1004

         -  templateId целое число ≥ 1 -> иначе 1006

8. Проверка уникальности дат

   -  Бек проверяет, что в items нет дублирующихся дат:

      -  Найдены дубликаты -> 1019 Ошибка валидации

9. Проверка существования офиса

   -  Бек проверяет существование офиса:

      -  Офис не найден -> 1013 Офис не найден

10. Проверка существования шаблонов

    -  Бек собирает все уникальные templateId из items

    -  Бек проверяет существование каждого шаблона в таблице templates:

       -  Шаблон не найден -> 1036 с указанием первого отсутствующего

11. Загрузка данных шаблонов

    -  Бек загружает template_shifts для всех найденных шаблонов

12. Начало транзакции

    -  Бек начинает транзакцию для обеспечения целостности данных

13. Удаление существующих смен

    -  Бек собирает все даты из items

    -  Бек удаляет из таблицы shifts все смены за указанные даты в указанном офисе

    -  Бек сохраняет количество удаленных смен

14. Применение шаблонов по дням

    -  Бек инициализирует пустой массив createdShifts

    -  Для каждого элемента items:

       -  Получает template_shifts для соответствующего templateId

       -  Для каждой templateShift:

          -  Создает amountWorkplaces вакантных смен (employeeId = NULL) с параметрами:

             -  officeId из запроса

             -  divisionId из templateShift

             -  startAt = [item.date](http://item.date) + startOn

             -  endAt = [item.date](http://item.date) + endOn

             -  status = 'draft'

          -  Добавляет созданные смены в массив createdShifts

15. Фиксация транзакции

    -  Бек фиксирует транзакцию (COMMIT)

    -  При возникновении ошибки на любом этапе - откат транзакции (ROLLBACK) и возврат ошибки

16. Формирование ответа

    -  Бек формирует объект ответа:

       -  shifts - массив созданных смен

       -  deletedCount - количество удаленных смен

    -  Бек возвращает статус 201 Created

17. Обработка на фронте

    -  Фронт обновляет отображение календаря с новыми сменами

    -  Фронт показывает уведомление об успешном применении шаблонов

    -  Фронт сбрасывает выделение дней

<table header="row">

<td>

```
@startuml
title Получение списка заявок на подмену (полная диаграмма)

participant "Frontend" as Front
participant "Schedule Service" as Schedule
participant "Office Service" as Office
participant "Employee Service" as Employee

== Отправка запроса ==
Front -> Schedule: GET /schedule/substitution-requests\nс query-параметрами

== Проверка авторизации ==
Schedule -> Schedule: Проверка токена

alt Токен недействителен или отсутствует
    Schedule --> Front: 1010 Не авторизован
else Токен действителен
    Schedule -> Schedule: Извлечение user_id, roles,\noffice_id, region_id
    
    == Проверка роли и обязательных параметров ==
    alt ROLE_EMPLOYEE
        Schedule --> Front: 1011 Доступ запрещен
    else ROLE_OFFICE_MANAGER
        alt officeId отсутствует
            Schedule --> Front: 1050 Не передан обязательный параметр
        else officeId != office_id из токена
            Schedule --> Front: 1011 Доступ запрещен
        else officeId совпадает
```

**Описание работы метода**

1\. Проверка авторизации

-  Бек проверяет подлинность токена через Auth Service

-  Если токен отсутствует, недействителен или истек -> ошибка 1010 "Не авторизован"

-  Если токен действителен -> извлекает `user_id`, `roles`, `office_id`, `region_id`

2\. Валидация тела запроса

| Условие                             | Ошибка                                                      |
|-------------------------------------|-------------------------------------------------------------|
| `requestId` отсутствует             | 1073 "Не заполнен обязательный параметр requestId"          |
| `requestId` \< 1 или не число       | 1074 "Недопустимое значение requestId"                      |
| `cancellationReason` отсутствует    | 1058 "Не заполнен обязательный параметр cancellationReason" |
| `cancellationReason` > 500 символов | 1055 "Превышена максимальная длина комментария"             |

3\. Получение заявки из БД

-  Выполняется запрос к таблице `substitution_requests` по `id = requestId`

-  Запрашиваются поля: `id`, `shift_id`, `status_id`, `is_urgent`, `comment`

-  Если заявка не найдена -> ошибка 1071 "Заявка на подмену не найдена"

4\. Получение информации о смене

-  Выполняется запрос к таблице `shifts` по `id = shift_id` из заявки

-  Запрашивается поле `office_id` для проверки прав доступа

-  Если смена не найдена (техническая ошибка, невозможная ситуация) -> ошибка 1022 "Смена не найдена"

5\. Проверка прав доступа

5\.1. Для руководителя офиса (`ROLE_OFFICE_MANAGER`):

-  Сравнивается `office_id` из токена с `office_id` смены

-  Если не совпадают -> ошибка 1011 "Доступ запрещен"

5\.2. Для территориального менеджера (`ROLE_TERRITORY_MANAGER`):

-  Запрашивается список офисов региона через Office Service по `region_id` из токена

-  При ошибке от Office Service -> 1063 "Ошибка при обращении к сервису Office"

-  Проверяется, что `office_id` смены входит в список офисов региона

-  Если не входит -> ошибка 1011 "Доступ запрещен"

5\.3. Для линейного сотрудника (`ROLE_EMPLOYEE`):

-  Доступ запрещен -> ошибка 1011 "Доступ запрещен"

6\. Проверка статуса заявки

| Текущий `status_id` | Результат                                       |
|---------------------|-------------------------------------------------|
| 3 (cancelled)       | Ошибка 1076 "Заявка уже отменена"               |
| 2 (assigned)        | Ошибка 1075 "Заявка не в статусе 'in_progress'" |
| 1 (in_progress)     | Можно отменять                                  |

7\. Обновление заявки

| Поле в БД             | Значение                   | Источник                          |
|-----------------------|----------------------------|-----------------------------------|
| `status_id`           | 3 (ID статуса "cancelled") | Из справочника `request_statuses` |
| `cancellation_reason` | `cancellationReason`       | Из тела запроса                   |

8\. Получение смежных данных заявки (RETURNING)





-  `shift_id` -- ID смены

-  `is_urgent` -- флаг срочности

-  `comment` -- комментарий к заявке





9\. Формирование ответа

| Поле ответа          | Значение             | Источник                |
|----------------------|----------------------|-------------------------|
| `id`                 | `requestId`          | Из тела запроса         |
| `shiftId`            | `shift_id`           | `substitution_requests` |
| `statusId`           | 3                    | Фиксированное значение  |
| `isUrgent`           | `is_urgent`          | Из RETURNING            |
| `comment`            | `comment`            | `substitution_requests` |
| `cancellationReason` | `cancellationReason` | Из тела запроса         |

12\. Получение контактов начальника офиса, get /employees/\{id}/contacts, поле email

13\. Получение ФИО начальника офиса, get /employees/\{id}, поля firstName, surname, lastName

14\. Отправление уведомлений о появлении заявки на подмену

-  EXCHANGE_NAME: notifications.notifications

-  EXCHANGE_TYPE: direct

-  ROUTING_KEY: notifications.event

   -  Body:

      -  type: substitutionSolved

      -  email: \{email начальника}

-  При успешном выполнении возвращается 200 OK с объектом заявки

-  Все операции выполняются в рамках одной транзакции

**Описание работы метода**

1\. Проверка авторизации

-  Бек проверяет подлинность токена через Auth Service

-  Если токен отсутствует, недействителен или истек -> ошибка 1010 "Не авторизован"

-  Если токен действителен -> извлекает `user_id`, `roles`, `region_id`

2\. Проверка роли

-  Только роль `ROLE_TERRITORY_MANAGER` может назначать сотрудников

-  Если роль `ROLE_OFFICE_MANAGER` или `ROLE_EMPLOYEE` -> ошибка 1011 "Доступ запрещен"

3\. Валидация тела запроса

| Условие                        | Ошибка                                              |
|--------------------------------|-----------------------------------------------------|
| `requestId` отсутствует        | 1073 "Не заполнен обязательный параметр requestId"  |
| `requestId` \< 1 или не число  | 1074 "Недопустимое значение requestId"              |
| `employeeId` отсутствует       | 1078 "Не заполнен обязательный параметр employeeId" |
| `employeeId` \< 1 или не число | 1079 "Недопустимое значение employeeId"             |

4\. Получение заявки из БД

-  Выполняется запрос к таблице `substitution_requests` по `id = requestId`

-  Запрашиваются поля: `id`, `shift_id`, `status_id`, `is_urgent`, `comment`

-  Если заявка не найдена -> ошибка 1071 "Заявка на подмену не найдена"

5\. Проверка статуса заявки

-  Заявка должна иметь статус `in_progress` (statusId = 1)

-  Если статус = 2 (assigned) или 3 (cancelled) -> ошибка 1075 "Заявка не в статусе 'in_progress'"

6\. Получение информации о смене

-  Выполняется запрос к таблице `shifts` по `id = shift_id` из заявки

-  Запрашиваются поля: `id`, `office_id`, `division_id`, `employee_id`, `start_on`, `end_on`

-  Если смена не найдена -> ошибка 1022 "Смена не найдена"

7\. Проверка принадлежности офиса к региону

-  Запрашивается список офисов региона через Office Service по `region_id` из токена

-  При ошибке от Office Service -> 1063 "Ошибка при обращении к сервису Office"

-  Проверяется, что `office_id` смены входит в список офисов региона

-  Если не входит -> ошибка 1011 "Доступ запрещен"

8\. Проверка существования сотрудника

-  Запрашивается в Employee Service информация о сотруднике по `employeeId`

-  При ошибке от Employee Service -> 1062 "Ошибка при обращении к сервису Employees"

-  Если сотрудник не найден -> ошибка 1012 "Сотрудник не найден"

9\. Получение данных сотрудника

-  Запрашиваются в Employee Service:

   -  `region_id` -- регион сотрудника

   -  `divisionId` -- направление

   -  `is_floating` -- флаг группы подмены

   -  ФИО сотрудника

-  При ошибке от Employee Service -> 1062 "Ошибка при обращении к сервису Employees"

10\. Проверка принадлежности сотрудника к региону

-  Сравнивается `region_id` сотрудника с `region_id` из токена территориального менеджера

-  Если регионы не совпадают -> ошибка 1011 "Доступ запрещен"

-  Важно: После подтверждения региона, сотрудник (любого типа) может работать в любом офисе этого региона

11\. Проверка соответствия направления

-  Сравнивается `divisionId` сотрудника с `division_id` смены

-  Если не соответствуют -> ошибка 1072 "Сотрудник не подходит по направлению"

12\. Проверка доступности сотрудника

-  Проверяется таблица `shifts` на наличие пересекающихся смен у сотрудника:

   -  `employee_id = :employeeId`

   -  дата смены совпадает с датой назначаемой смены

   -  интервалы времени пересекаются

   -  статус смены не `cancelled`

-  Если есть пересечение -> ошибка 1019 "Конфликт с состоянием сервера"

13\. Проверка блокировок

-  Запрашивается в Employee Service наличие активных блокировок (отпуск/больничный) на дату смены

-  При ошибке от Employee Service -> 1062 "Ошибка при обращении к сервису Employees"

-  Если есть активная блокировка -> ошибка 1045 "Сотрудник заблокирован на указанную дату"

14\. Назначение сотрудника на смену

-  Обновляется запись в таблице `shifts`:

   -  `employee_id = :employeeId`

15\. Обновление статуса заявки

| Поле в БД   | Значение                  | Источник                          |
|-------------|---------------------------|-----------------------------------|
| `status_id` | 2 (ID статуса "assigned") | Из справочника `request_statuses` |

16\. Получение информации о сотруднике

-  Запрашивается в Employee Service ФИО сотрудника для ответа

-  При ошибке от Employee Service -> 1062 "Ошибка при обращении к сервису Employees"

17\. Формирование ответа

| Поле ответа                                         | Значение              | Источник                   |
|-----------------------------------------------------|-----------------------|----------------------------|
| `id`                                                | `requestId`           | Из тела запроса            |
| `shiftId`                                           | `shift_id` из заявки  | Из `substitution_requests` |
| `statusId`                                          | 2                     | Фиксированное значение     |
| `isUrgent`                                          | `is_urgent` из заявки | Из `substitution_requests` |
| `comment`                                           | `comment` из заявки   | Из `substitution_requests` |
| [`assignedEmployee.id`](http://assignedEmployee.id) | `employeeId`          | Из тела запроса            |
| `assignedEmployee.lastName`                         | Фамилия сотрудника    | Из Employee Service        |
| `assignedEmployee.firstName`                        | Имя сотрудника        | Из Employee Service        |
| `assignedEmployee.surname`                          | Отчество сотрудника   | Из Employee Service        |

18\. Возврат ответа

-  При успешном выполнении возвращается 200 OK с объектом заявки

**Описание работы метода**

**1\.** Проверка авторизации

-  Бек проверяет подлинность токена через Auth Service

-  Если токен отсутствует, недействителен или истек -> ошибка 1010 "Не авторизован"

-  Если токен действителен -> извлекает `user_id`, `roles`, `office_id`

2\. Валидация path параметра

| Условие                     | Ошибка                                         |
|-----------------------------|------------------------------------------------|
| `shiftId` \< 1 или не число | 1024 "Указано недопустимое значение в shiftId" |

3\. Получение информации о смене

-  Выполняется запрос к таблице `shifts` по `id = shiftId`

-  Запрашиваются поля: `id`, `office_id`, `employee_id`

-  Если смена не найдена -> ошибка 1022 "Смена не найдена"

4\. Проверка прав доступа

-  Сравнивается `office_id` из токена с `office_id` смены

-  Если не совпадают -> ошибка 1011 "Доступ запрещен"

5\. Получение заявки из БД

-  Выполняется запрос к таблице `substitution_requests` по `shift_id = shiftId`

-  Запрашиваются поля: `id`, `shift_id`, `status_id`, `is_urgent`, `comment`, `cancellation_reason`

6\. Формирование ответа (случай А: заявка не найдена)

-  Если заявка не найдена, возвращается 200 OK с телом `null`

7\. Формирование ответа (случай Б: заявка найдена)

-  Если заявка найдена, формируется объект заявки:

| Поле ответа          | Значение              | Источник                           |
|----------------------|-----------------------|------------------------------------|
| `id`                 | `id`                  | Из таблицы `substitution_requests` |
| `shiftId`            | `shift_id`            | Из таблицы `substitution_requests` |
| `statusId`           | `status_id`           | Из таблицы `substitution_requests` |
| `isUrgent`           | `is_urgent`           | Из таблицы `substitution_requests` |
| `comment`            | `comment`             | Из таблицы `substitution_requests` |
| `cancellationReason` | `cancellation_reason` | Из таблицы `substitution_requests` |

8\. Обогащение данных для статуса assigned

-  Если `status_id = 2` (assigned):

   -  Получается `employee_id` из связанной смены (уже получен на шаге 3)

   -  Запрашивается в Employee Service ФИО сотрудника по `employee_id`

   -  При ошибке от Employee Service -> 1062 "Ошибка при обращении к сервису Employees"

   -  Добавляется объект `assignedEmployee` в ответ

9\. Возврат ответа

-  При успешном выполнении возвращается 200 OK с объектом заявки или `null`
